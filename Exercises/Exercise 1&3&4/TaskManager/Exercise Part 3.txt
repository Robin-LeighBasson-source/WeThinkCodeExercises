1. Test Knowledge Answers
Q1 (Conflicting Signals): A MEDIUM task (Priority 2) due today would outrank an URGENT task (Priority 4) due in 3 weeks. The system calculates a dynamic importance score: the medium task gets a massive point boost for the imminent deadline, while the urgent future task gets no deadline bonus. The system works this way to prioritize immediacy and momentum, ensuring users don't miss imminent deadlines while focusing on distant goals.

Q2 (Status Design Philosophy): The REVIEW status solves the business problem of "external dependency" or "quality control." It accurately models the domain reality that a task is out of the active worker's hands and is awaiting approval, which prevents the task from artificially inflating the "In Progress" workload.

Q3 (The Tag Pattern): A user sets priority to URGENT when they personally feel a task is highly important (emotional intent). They use a "blocker" tag to indicate a structural reality: other tasks cannot proceed until this is finished (technical dependency). The domain model uses tags to inject specific contextual weights (+8 score) into the algorithm to separate intent from reality.

Q4 (Sync Implications): If a task is updated on a laptop and then a phone before syncing, the system uses the updatedAt timestamp to resolve the conflict. This is a classic distributed system pattern called "Last Write Wins" (LWW), making updatedAt the ultimate source of truth for temporal consistency.

Q5 (Statistics & Domain Insights): Counting "completed in the last 7 days" measures velocity and recent productivity. An all-time completion count is a vanity metric that loses meaning over time. The 7-day window gives the user a realistic sense of their current momentum.

TASK (Core Entity)
├─ Identity
│  └─ id: UUID
├─ Content
│  ├─ title: string
│  ├─ description: string
│  └─ tags: string[]
├─ Workflow State
│  └─ status: TODO | IN_PROGRESS | REVIEW | DONE
├─ Timing (Temporal Dimensions)
│  ├─ createdAt: Date
│  ├─ updatedAt: Date
│  ├─ completedAt: Date (if DONE)
│  └─ dueDate: Date (optional)
├─ User-Set Priority (STATIC)
│  └─ priority: 1|2|3|4
└─ System-Calculated Importance (DYNAMIC)
   ├─ Base: priority * 10
   ├─ +/- Due date factor
   ├─ +/- Status factor
   ├─ +/- Tag factor
   └─ +/- Recency factor
   = Importance Score
            ↓
       [Used for sorting/ranking]
3. Domain Glossary
Task Lifecycle: The strict, four-step workflow pipeline a unit of work must travel through (TODO, IN_PROGRESS, REVIEW, DONE).

Static Priority: The explicit, user-defined urgency level (1-4) assigned at creation. Represents human intent.

Dynamic Importance Score: A system-calculated, real-time value that ranks tasks by combining Static Priority with temporal context (deadlines, recency) and technical blockers (tags).

Temporal Business Logic: The system's awareness of time, utilizing dueDate for urgency routing, updatedAt for conflict resolution during syncing, and completedAt for velocity metrics.
