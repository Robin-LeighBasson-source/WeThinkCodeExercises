1. My initial understanding vs. what I discovered

Initial: I assumed the task prioritization system simply read the 1-4 priority integer from the tasks.json file and sorted based on that single number.

Discovered: Through guided questioning with the AI, I discovered that relying on a static number is flawed. I learned how to architect a Dynamic Importance Score—a system that calculates a real-time score on the fly by combining the user's base priority with temporal context (due dates, recency) and system states (DONE circuit breakers, blocker tags).

2. The key insights the guided questions helped me uncover

Derived State: The AI guided me to realize that calculated values (like dynamic scores) should never be saved to the database (JSON file). They should be calculated on the fly during the read operation to prevent state de-synchronization.

Architectural Separation: I learned that the CLI layer should never import scoring utilities directly. The CLI should pass the --sort flag to the Business Logic layer (app.js), which then orchestrates the sorting using the Strategy Pattern.

Performance Optimization: I was introduced to the Schwartzian Transform (Map-Sort-Map) to prevent the calculateScore function from being unnecessarily recalculated hundreds of times during a .sort() operation.

3. Misconceptions I had that were clarified

The DONE Edge Case: I initially thought a DONE task should just immediately return a massive penalty score. The AI's questions helped me realize that this would bury a task I completed 30 seconds ago. I clarified my logic to apply a recency bonus first, so recently finished tasks still show up temporarily to give a sense of momentum.

Weighting Balance: I initially made the base priority multiplier too small. The AI showed me scenarios where an overdue LOW task would outrank a non-overdue URGENT task. I fixed this by heavily increasing the weight of the base priority (Base x 20) to ensure user intent wasn't entirely overwritten by deadlines.
