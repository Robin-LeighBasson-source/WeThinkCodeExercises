1. Data Flow Diagram
The completion flow travels through five distinct stages, ensuring that user intent is transformed into a persistent, valid system state:

CLI Layer (cli.js): Intercepts raw user input (status abc123 done).

Controller Layer (app.js): Coordinates the lookup. It finds the task in the cache and identifies the "Done" intent.

Domain Model (models.js): Executes the markAsDone() method. This is where the Semantic State Change occurs—mutating the status and stamping the completedAt time.

Cache Layer (storage.js): Holds the updated object in memory.

Persistence Layer (tasks.json): Serializes the entire state to a string and overwrites the file.

2. State Changes Occurring
The transition is Atomic. When a task is marked done, the following invariants are updated:

status: todo ➔ done

completedAt: null ➔ Date.now()

updatedAt: Syncs to completedAt

isOverdue() result: Transitions to false (Logic: Completed tasks cannot be overdue).

3. Potential Points of Failure

ID Collision/Missing: If the CLI passes an invalid ID, the chain breaks at the Controller layer with no state change.

Serialization Drift: If the JSON.parse logic in the storage layer fails to convert the "done" date string back into a Date object, future statistics will crash.

The "Vulnerable Window": There is a millisecond gap between the model changing its status in memory and the storage layer finishing the write to disk. If the app crashes here, the task is "done" in the user's mind but "todo" in the file.

4. Persistence Mechanism
The app uses Full-State Overwrite. Rather than updating one line, it persists the entire in-memory "source of truth" to the disk. This ensures that the tasks.json file is always a perfect snapshot of the current application state.
